Contrairement à Bind qui fait tout, unbound fait peu mais bien . [2].

nsd :

  • Il fait autorité sur une zone, et ne joue pas le rôle de serveur récursif.
  • Attention, nsd ne gère pas les views,split-views,split-horizon,... de bind; soit le fait de répondre différement à des requêtes en fonction de l'adresse source.Il va falloir ruser pour gérer le LAN.

unbound :

  • récursif
  • peut aussi servir des zones locales


C'est plus clair, plus sécurisé, mais moins classique, habitués que nous sommes à Bind..

unbound, le résolveur, juste résolveur

Config

 pkg_add unbound
 echo '    syslogd_flags="${syslogd_flags} -a /var/unbound/dev/log" ' >> /etc/rc.conf.local
 unbound-control-setup
touch /var/log/unbound.log

Modifier /var/unbound/etc/unbound.conf 

interface: 0.0.0.0
interface: ::0
access-control: 10.0.0.0/24 allow
logfile: "/var/log/unbound.log"
hide-identity: yes  
hide-version: yes  
forward-zone: # A qui forwarder les requêtes dont les réponses n'ont pas encore été cachées..
    name: "."
    forward-addr: 212.27.40.240  # serveur DNS Free
    forward-addr: 8.8.8.8        # serveur DNS Google...qu'on peu oublier


On peut aussi demander à Unbound de servir une zone locale. 

private-domain: "MonLAN" 
local-zone: "MonLAN." static                      #On n'oublie pas le point final!!
local-data: "PC1.MonLAN. IN A IP1"
local-data: "PC2.MonLAN. IN A IP2"
local-data-ptr: "IP1 PC1.MonLAN"
local-data-ptr: "IP2 PC2.MonLAN"

Le debug se fait bien avec : 

unbound -dvvvvv

Attention si vous voulez fournir de la résolution à de multiples plages disjointes, il faut répéter les directives : 

access-control: 10.20.0.0/24 allow
access-control: 10.100.0.0/24 allow

Nsd, le serveur faisant autorité

Config

nsd est installé par défaut.

Vous devez d'abord générer le certificat permettant le contrôle de nsd: 

nsd-control-setup

La syntaxe est bind-compliant; c'est toujours ça. 

/var/nsd/etc/nsd.conf

server:
       hide-version: yes
logfile: "/var/log/nsd.log"
verbosity: 1
ip4-only: yes
zonesdir: "/var/nsd/zones"
remote-control:
## master zone example
zone:
       name: "openbsd-edu.net"
       zonefile: "openbsd-edu"
       notify: IP_Secondaire NOKEY
       notify:  IP_Secondaire_2 NOKEY
       provide-xfr:  IP_Secondaire NOKEY
       provide-xfr:  IP_Secondaire_2 NOKEY
## slave zone example
zone:
       name: "openbsd-edu.net"
       zonefile: "openbsd-edu"
       allow-notify:: IP_Maitre NOKEY
       request-xfr:  IP_Maitre  NOKEY


Et la classique séance désormais : 

rcctl enable nsd
rcctl start nsd


C'est tout !

Pour changer le port d'écoute (si unbound tourne sur la même machine) 

port: 5345

N'oubliez pas : Deux points espace Alors, pour mémoire, nsd n'est pas récursif. Donc un dig @ donnera 

;; WARNING: recursion requested but not available

mais une réponse si la requête porte sur une machine dans la zone, et rien dans le cas contraire.


Ben, oui !

echo 'named_flags="-c /etc/named-dual.conf -u named -t /var/named" ' >> /etc/rc.conf.local

Puis vous configurez vos zones comme vous l'entendez.

SPF

  • Un enregistrement (txt ou spf) permettant d'afficher aux autres MTA quelles sont vos machines autorisées à servir de SMTP sortant du domaine. Les serveurs SMTP utilisant SPF récupéreront la liste de vos SMTP et autoriseront ou pas la réception de mails.

A ajouter à /var/named/master/db.votrezone : 

openbsd-edu.net. IN      TXT "v=spf1 mx ptr ip4:82.237.239.48  ip6:2a01:e35:2ede:f300:201:2ff:fe9d:daee -all"

Qui signifie : Je déclare que :

  • Tous les MX du domaine,
  • tous les enregistrements PTR du domaine,
  • l'adresse IPv4 :machin,
  • l'adresse IPv6 : truc,
  • et eux seulement

sont autorisés à expédier des mails ! Tout autre serveur SMTP déclarant envoyer des mails dont le champ From est openbsd-edu.net est un menteur et il faut refuser ses mails.

Remarque: Un serveur SMTP validant SPF recevant un mail d'un domaine sans enregistrement SPF a une compatibilité ascendante et accepte par défaut le mail.

Debug

La zone n'est plus trouvée dans le résolveur local

Et dans les logs  : 

zone  MAZONE loading from master file master FILE failed: ran out of space

En fait, les enregistrements TXT ne doivent pas dépasser 255 caractères, et si vous mettez des enregistrements SPF avec de l'IPv6, vous l'avez dans le baba !
